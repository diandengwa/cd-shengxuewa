## 一、项目背景与现状诊断

### 1.1 项目定位

点灯蛙（www.diandengwa.com）是面向成都K12家长的升学政策智能诊断系统，核心价值主张："**输入你家情况，输出个性化升学路线图**"。部署在腾讯云（162.14.74.251），已完成ICP备案和公安联网备案。

### 1.2 后端现状（已生产就绪）

| 模块 | 状态 | 说明 |
|------|------|------|
| 四步裁决引擎 | ✅ 生产运行 | 情况理解→灰色地带→竞争烈度→时间线，Step2-4并行执行 |
| 政策Wiki | ✅ 已部署 | 154页政策数据，覆盖成都23区县3年政策 |

## 三、顶层品牌架构设计

### 3.1 三条功能线的本质分析

点灯蛙品牌旗下规划了三条功能线，服务同一人群（成都K12家长）但处于不同阶段、不同场景：

| 维度 | ① 成都K12升学参谋 | ② 智能学伴+提分 | ③ 成都养虾教练 |
|------|------|------|------|
| **核心价值** | 信息不对称→确定性 | 学习效率→成绩提升 | 技能赋能→自主掌控 |
| **目标用户** | 焦虑期家长（升学决策时刻） | 家长+学生（日常学习） | 愿意折腾的技术型家长 |
| **使用频率** | 季节性（每年1-3次） | 高频（每日/每周） | 前重后轻 |
| **技术底座** | 政策Wiki+四步引擎（**已就绪**） | 提示词库+学习算法（**未建成**） | OpenClaw/Agent指南（**已文档化**） |
| **变现模式** | ¥99/月订阅 | ¥29.9-99/月 | ¥399-2999课程/陪跑 |
| **用户心智** | "帮我判断行不行" | "帮我孩子提分" | "教我自己养一只AI" |
| **去AI化要求** | 严格（专家定位） | 适度（"智能"可接受） | 不限制（AI是教学对象） |
| **成熟度** | ★★★★★ 可上线 | ★★☆☆☆ 概念阶段 | ★★★☆☆ 方案完整待产品化 |

### 3.2 推荐架构：伞形品牌 + 单产品先行

三条线不是并列关系，而是**用户生命周期递进关系**：

```
家长焦虑升学 → 用升学参谋（¥99/月）
     ↓ 孩子入学后
日常学习需求 → 用智能学伴（¥29.9/月）
     ↓ 部分深度用户想自己掌控
自主赋能需求 → 学养虾教练（¥399+课程）
```

**架构设计**：

```
diandengwa.com（品牌门户 = 着陆页）
│
├── /app    升学参谋（Phase 1，现在做，核心收入）
│           定位：成都K12升学政策研判专家
│           口吻：严肃专业，去AI化
│
├── /companion  智能学伴（Phase 2，3-6个月后）
│               定位：陪伴孩子日常学习的智能伙伴
│               口吻：温暖亲和，"智能"可用
│
├── /coach  养虾教练（Phase 3，6-12个月后）
│           定位：教家长自己搭建和调教AI学伴
│           口吻：极客导师，AI Agent是主角
│
└── /pay    统一支付页（三个产品共用）
```

### 3.3 架构决策理由

1. **品牌统一但产品分离**——升学决策严肃，养虾极客，混在一个界面互相稀释。但共享"点灯蛙"IP：升学参谋是"蛙蛙帮你判断"，智能学伴是"蛙蛙陪你学习"，养虾教练是"蛙蛙教你养一只属于自己的蛙"

2. **着陆页承担品牌门户角色**——展示点灯蛙完整使命"从升学决策到日常学习到自主赋能"，三个产品线像三张卡片，当前只有升学参谋可点击，另外两个标注"即将上线"

3. **升学参谋先行**——后端四步引擎已生产部署、政策Wiki已有154页、支付已跑通。先做到极致验证商业模式，再扩展

4. **"成都养虾"谐音是社群资产，不是产品名**——"养虾=养蛙=养点灯蛙"是社群内部文化暗号，不应出现在面向公众的产品名里。对外叫"点灯蛙·自主赋能营"

### 3.4 点灯蛙角色定位

| 维度 | 定位 |
|------|------|
| 身份 | 成都升学参谋（当前阶段） / 智能学伴（品牌品类词） |
| 职能 | 升学政策研判与路径规划 |
| 口吻 | "我是你的升学参谋" |
| 信任背书 | 专家智库 + 政策数据库 + 多年升学案例积累 |
| Slogan | 不是代你学，是启你悟（保留不变） |

## 四、技术架构决策

### 4.1 技术栈选择：Alpine.js + 原生CSS

**决策**：采用Alpine.js（方案B），不使用纯原生JS，也不使用Vue/React。

**选择理由**：
- Alpine.js被称为"现代版jQuery"，提供Vue式响应式数据绑定，但零构建依赖
- 仅引入一个15KB的文件（CDN或本地托管），不改变FastAPI `FileResponse` + `StaticFiles`架构
- 未来扩展用户注册登录时，`Alpine.store()`天生解决全局状态同步
- 表单密集型场景（注册、家庭画像、支付）是Alpine强项，`x-model`双向绑定省大量手写代码
- 退出成本低：Alpine模板语法和Vue几乎一样，将来迁移Vue/React是平滑的

### 4.2 前端架构选择：混合架构（方案C）

**决策**：着陆页独立 + 功能工作台单页 + 支付页独立。

```
GET /        → static/index.html     (着陆页/品牌门户)
GET /app     → static/app.html       (升学参谋工作台SPA)
GET /pay     → static/pay.html       (支付页)
GET /static/* → 静态资源 (CSS/JS/图片)
```

**选择理由**：
1. 着陆页独立——首屏最快，营销转化不受功能逻辑拖累
2. 工作台内切换零延迟——对话/政策/个人中心状态互通，`Alpine.store()`共享
3. 支付页独立——微信支付涉及页面跳转和回调，独立处理最干净
4. 功能扩展时只改工作台，着陆页稳定不动
5. 部署不变——FastAPI `FileResponse` + `StaticFiles`，CI/CD流程不动

### 4.3 前后端同仓库

前端和后端放在同一Git仓库，前端文件纳入版本控制。项目体量不大（核心3-4个页面），引入构建链得不偿失。

### 6.3 Tab1 · 诊断（核心功能）

**阶段一：对话输入区**

- 学段选择器（幼升小/小升初/随迁入学/中考）胶囊标签
- 对话区域：AI消息左对齐（IP头像），用户消息右对齐（蜀都青气泡）
- AI主动给出结构化选项卡（替代纯文本输入），降低家长表达门槛
- 快捷提问按钮（可折叠）
- 输入框固定底部 + 配额提示 + 深度诊断CTA

### 7.2 核心用户流程

**流程1：首次访问 → 免费诊断**
```
着陆页 → [立即免费诊断] → 工作台检测未登录 → 微信OAuth授权
→ 回调携带redirect=/app&tab=diagnosis → 选择学段 → 开始对话
→ 免费基础8段输出 → [深度诊断] → 灯泡进度动画 → 四步裁决结果
```
- OAuth token存入localStorage，后续请求自动携带
- OAuth回调时携带redirect参数，不中断对话流

**流程2：免费额度用尽 → 付费转化**
```
诊断中配额检查剩余0次 → 显示付费墙弹窗 → [立即升级] → /pay
→ 选择LITE → 微信支付¥29.9 → 回调验证 → 自动生成邀请码
→ 自动升级套餐 → redirect→/app → 显示庆祝动画 → 继续诊断
```

**流程3：深度诊断进行中**
```
POST /api/diagnose → 4灯泡进度动画 → 后端并行执行四步(asyncio.gather)
→ 逐步返回结果(每完成一步渲染) → 全部完成 → 行动清单生成 → IP切换灵感态
```

### 7.3 错误处理与降级策略

| 场景 | 处理方式 |
|------|----------|
| 微信OAuth失败 | "授权失败，请重试" + 游客模式仅浏览政策工具 |
| 诊断API超时(>90秒) | "研判超时，请稍后重试"，保留对话历史 |
| 诊断API报错 | 参谋疲惫态IP + "我需要缓一缓" |
| 支付回调验证失败 | "支付确认中，请勿重复支付"，后台异步轮询 |
| 网络断开 | 全局监听offline，顶部蜀都青提示条 |

### 7.4 数据持久化策略

| 数据 | 存储位置 | 生命周期 |
|------|----------|----------|
| 登录token | localStorage | 7天过期自动刷新 |
| 对话历史 | localStorage + 后端 | 本地即时，后端持久化 |
| 家庭画像 | 后端SQLite | 永久，登录后同步本地 |
| 诊断报告 | 后端SQLite | 永久，历史记录列表读取 |
| 行动清单勾选 | localStorage | 本地持久 |
| 当前Tab状态 | sessionStorage | 关闭浏览器清除 |

### 7.5 性能优化要点

| 优化项 | 方案 |
|--------|------|
| 首屏加载 | 着陆页纯静态<50KB；工作台Alpine(15KB)+CSS+JS<100KB |
| 图片优化 | IP形象WebP格式，LCP图<80KB；备案图标内联SVG |
| 诊断结果渲染 | 逐步渲染，不等全部完成 |
| API请求 | 对话消息合并提交 |

| AI辅助 | 专家辅助 / 参谋把关 | |

### 8.3 角色重新定位

| 维度 | 原定位（禁用） | 新定位 |
|------|------|------|
| 身份 | 家庭教育AI学伴 | 成都升学参谋 |

## 十二、待讨论的开放问题

### 12.1 定价策略

当前后端代码中 LITE ¥29.9/月，但 `完整方案_v1.0.md` 建议取消¥29.9直接从¥99起步。两种方案：

- **B. 取消¥29.9**：FREE → PRO ¥99/月 → VIP ¥999/次（高客单价专业定位）

### 12.2 着陆页定位

- **A. 品牌门户**：展示三条产品线，升学参谋为当前主推，另外两个"即将上线"（长期品牌建设）
- **B. 单产品落地页**：只推升学参谋，不暴露未来规划（短期转化最大化）

### 12.3 政策工具深度

- **A. 静态数据展示**（当前方案）：日历/黑话/划片用预存数据，不做实时交互
- **B. 接入后端Wiki**：日历和黑话调用后端API动态获取，划片接教育局数据

### 12.4 微信小程序 vs H5

当前方案为H5优先。是否需要在Phase 1同步规划小程序版本？小程序优势是微信生态内体验更好，劣势是审核周期长、迭代慢。

### 12.5 IP形象的AI生成

当前IP形象已有3D彩色版和黑白剪影版。四态表情（专注/灵感/疲惫/庆祝）需要按VIS v4.0的Prompt生成。是否在Phase 1就完成，还是先用简化版（emoji/图标）替代？

### 12.6 "养虾"概念在升学参谋中的出现程度

升学参谋产品本身是否需要提及"养虾"概念？
- **A. 完全不提**：升学参谋是独立专业产品，养虾是Phase 3的事
